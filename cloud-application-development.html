Cloud Computing Overview

Cloud Computing: the delivery of on-demand system resources, particularly data storage and computing power, without direct management by the user

  • large clouds often provide functions distributed over several servers
  • clouds can belong to an organisation or be public
  • cloud computing relies on the sharing of resources to achieve coherence and scale
  • cloud computing reduces infrastructure costs
  • cloud computing improves speed, performance, and management

A Cloud Service Provider is a third-party company offering a cloud-based platform, infrastructure, application, or storage service. Examples of CSPs are Amazon Web Services (AWS), Microsoft Azure, and Google Cloud Platform.

  • Cloud Providers need to:
  • enable self-service access
  • manage the infrastructure the cloud runs on
  • The Cloud needs to meet requirements such as:
  • virtualisation
  • automation
  • standardisation
  • service management

Serverless Architectures

Serverless: cloud computing execution model where the cloud provider runs the server and dynamically manages resource allocation

  • abstraction of servers
  • event-driven applications
  • instant and automatic scalability

Serverless means that code runs only on-demand and on a per-request basis.

  • API requests are sent by consumers
  • Objects are retrieved, created, or moved in storage
  • Events are scheduled

Frontend: frontend infrastructure includes everything the end user interacts with, and makes up the user interface. This includes components such as local networks, web browsers, and web applications

  • User Interface - refers to all components the end user can access, to send requests or perform actions
  • Software - runs on the end user's machine and is comprised of client-side applications and browsers
  • Network - the end user's machine. This can be mobile or PC and should work regardless of computing power

Backend: backend architecture is comprised of hardware and storage located on servers, and powers the frontend architecture

  • Application - manages requests and requirements made by the user and provides functionality to the user interface
  • Service - manages resources on servers such as tasks, storage, and computing
  • Runtime - the virtual environment where services such as apps, servers, storage, and networks are run
  • Storage - location where data is stored. This can be a HDD, SSD, flash drive, RAID storage, etc. Data is partitioned in storage for accessibility and redundancy
  • Infrastructure - engine managing all the services, which has a CPU, GPU, motherboard, network cards, input and output
  • Management - middleware which allocates resources to specific tasks and communicates between frontend and backend
  • Security - safety of data via redundancy and backups, as well as safety of networks via firewalls and protocols

new Date().getFullYear()

 alert("correct");

else alert("incorrect");

 }

 Callbacks & Promises
  
 Callback: function which executes after another function is complete

  • a function passed into another function is called a callback
  • callbacks are executed after other functions are complete
  • functions which accept callbacks as parameters are called higher-order functions
  • callbacks ensure code executes in the correct order


Callback declaration:

 function addOne(number, callback) {

 alert(number +1);

 callback();

 }

 addOne(1, callback);


 Promise: alternative to a callback which represents the result of an asynchronous operation

 A promise can be pending, fulfilled, or rejected:

  • Pending: state showing the initial state of a promise
  • Fulfilled: state showing a successful operation
  • Rejected: state showing a failed operation

Promise declaration:

  • the function passed to the new promise is the executor
  • the executor contains producing code which outputs a result
  • the executor starts automatically on creation

     promise = new Promise(function(resolve, reject) {

     statement;

     });

      
  • resolve(value) indicates the job has completed successfully
  • sets the state to "fulfilled"
  • sets the result to value
  • reject(error) indicates an error has occurred
  • sets the state to "rejected"
  • sets the result to error


 

Promises can be connected to consuming functions such as then and catch:

  • Then function
  • function(result) runs when the promise is resolved
  • and receives the result
  • function(error) runs when the promise is rejected
  • and receives the error

     promise.then(function(result) { statement; }, function(error) { statement; });
  • Catch function
  • catch is used if you're only interested in errors
  • is a shorthand method of omitting the result function

     promise.catch(alert);

     OR

     promise.then(null,alert);
  • to omit the error function, promise.then can be shortened

     promise.then(alert);

     OR

     promise.then(alert,null);
     

Software Development Practices and Tips
 
 Practices for Software Development

 What are the barriers to writing good code?

  • functional goals often take precedence
  • a lot of software development is self-taught
  • progress can be slow when developing applications
  • wide variety of frameworks and practices

Technical Debt: implied cost of additional work caused by choosing an limited solution rather than a more complex approach

  • like financial debt, code gathers interest over time
  • debt is paid off in maintenance
  • if not maintained, software is unable to evolve

Maintainability: ease with which a software system or component can be modified to correct faults, improve performance, or adapt to a changed environment

  • adding new features, without introducing bugs
  • fixing bugs, without introducing new bugs
  • improvement in performance and usability
  • improvement in productivity
  • support for multiple environments
  • easier collaboration

Top Tips for Software Development

  • Get the requirements right
  • Decide what you are building and who it is for
  • Most errors are introduced during requirements analysis and design
  • Leaving errors in code reduces maintainability and increases technical debt
  • Write code assuming others will see it
  • Use comments in code
  • Use consistent-looking code
  • Use code analysis tools e.g. pycodestyle, pylint, flake8
  • Write code assuming others will run it
  • Provide an automated build process
  • Provide a test suite
  • Automate install, build, and test together
  • Provide infrastructure support
  • run and install overnight
  • email reporting
  • continuous integration
  • Get others to check code
  • Collaborate with other team members
  • Focus on critical sections of code
  • Use version control
  • Use GitHub for version control
  • allows easy backups and commits
  • store code, documentation, unit tests, test data, custom scripts
  • improves collaborative development
